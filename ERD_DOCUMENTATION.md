## 1. COMPLETE ERD WITH NINE (9) TABLES (3 Marks)

### Tables Overview:

1. **provinces** - Top level of Rwanda administrative structure
2. **districts** - Second level administrative division
3. **sectors** - Third level administrative division
4. **cells** - Fourth level administrative division
5. **villages** - Fifth level (lowest) administrative division
6. **users** - Bus system users (passengers, drivers, admins)
7. **driver_licenses** - Driver license information
8. **routes** - Bus routes
9. **user_routes** - Join table for user-route bookings

---

## 2. DETAILED TABLE STRUCTURES AND RELATIONSHIPS

### Table 1: provinces
```sql
CREATE TABLE provinces (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    code VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL
);
```

**Purpose:** Stores Rwanda's 5 provinces (Kigali City, Southern, Western, Northern, Eastern)

**Relationships:**
- ONE province HAS MANY districts (One-to-Many)
- Foreign key stored in: districts.province_id

---

### Table 2: districts
```sql
CREATE TABLE districts (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    code VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    province_id BIGINT NOT NULL,
    FOREIGN KEY (province_id) REFERENCES provinces(id)
);
```

**Purpose:** Stores Rwanda's 30 districts

**Relationships:**
- MANY districts BELONG TO one province (Many-to-One)
  - Foreign Key: province_id → provinces.id
- ONE district HAS MANY sectors (One-to-Many)
  - Foreign key stored in: sectors.district_id

---

### Table 3: sectors
```sql
CREATE TABLE sectors (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    code VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    district_id BIGINT NOT NULL,
    FOREIGN KEY (district_id) REFERENCES districts(id)
);
```

**Purpose:** Stores Rwanda's 416 sectors

**Relationships:**
- MANY sectors BELONG TO one district (Many-to-One)
  - Foreign Key: district_id → districts.id
- ONE sector HAS MANY cells (One-to-Many)
  - Foreign key stored in: cells.sector_id

---

### Table 4: cells
```sql
CREATE TABLE cells (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    code VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    sector_id BIGINT NOT NULL,
    FOREIGN KEY (sector_id) REFERENCES sectors(id)
);
```

**Purpose:** Stores Rwanda's 2,148 cells

**Relationships:**
- MANY cells BELONG TO one sector (Many-to-One)
  - Foreign Key: sector_id → sectors.id
- ONE cell HAS MANY villages (One-to-Many)
  - Foreign key stored in: villages.cell_id

---

### Table 5: villages
```sql
CREATE TABLE villages (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    code VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    cell_id BIGINT NOT NULL,
    FOREIGN KEY (cell_id) REFERENCES cells(id)
);
```

**Purpose:** Stores Rwanda's 14,837 villages (sample in our database)

**Relationships:**
- MANY villages BELONG TO one cell (Many-to-One)
  - Foreign Key: cell_id → cells.id
- ONE village HAS MANY users (One-to-Many)
  - Foreign key stored in: users.village_id

**CRITICAL IMPORTANCE:** This is where users are linked to the administrative structure!

---

### Table 6: users
```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    village_id BIGINT NOT NULL,
    email VARCHAR(255) NOT NULL UNIQUE,
    name VARCHAR(255) NOT NULL,
    phone VARCHAR(255),
    user_type ENUM('ADMIN','DRIVER','PASSENGER') NOT NULL,
    FOREIGN KEY (village_id) REFERENCES villages(id)
);
```

**Purpose:** Stores all bus system users (passengers, drivers, administrators)

**Relationships:**
1. MANY users BELONG TO one village (Many-to-One)
   - Foreign Key: village_id → villages.id
   
2. ONE user HAS ONE driver_license (One-to-One)
   - Foreign key stored in: driver_licenses.user_id
   
3. ONE user CAN BOOK MANY routes (Many-to-Many through user_routes)
   - Join table: user_routes contains user_id

**KEY DESIGN DECISION:** Users are linked ONLY to village, not directly to province!
- This allows automatic traversal: User → Village → Cell → Sector → District → Province
- We can retrieve users by ANY administrative level through relationship chains

---

### Table 7: driver_licenses
```sql
CREATE TABLE driver_licenses (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    license_number VARCHAR(255) NOT NULL UNIQUE,
    expiry_date DATE NOT NULL,
    type VARCHAR(255) NOT NULL,
    user_id BIGINT NOT NULL UNIQUE,
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**Purpose:** Stores driver license information for drivers only

**Relationships:**
- ONE driver_license BELONGS TO one user (One-to-One)
  - Foreign Key: user_id → users.id
  - UNIQUE constraint ensures true one-to-one relationship

**Cascade Behavior:** When a user is deleted, their driver license is automatically deleted (orphanRemoval = true)

---

### Table 8: routes
```sql
CREATE TABLE routes (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    route_number VARCHAR(255) NOT NULL UNIQUE,
    start_location VARCHAR(255) NOT NULL,
    end_location VARCHAR(255) NOT NULL,
    distance DOUBLE,
    estimated_duration DOUBLE
);
```

**Purpose:** Stores bus route information

**Relationships:**
- ONE route CAN BE BOOKED BY MANY users (Many-to-Many through user_routes)
  - Join table: user_routes contains route_id

---

### Table 9: user_routes (JOIN TABLE)
```sql
CREATE TABLE user_routes (
    id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,
    booking_date TIMESTAMP(6) NOT NULL,
    route_id BIGINT NOT NULL,
    seat_number VARCHAR(255),
    status TINYINT CHECK (status BETWEEN 0 AND 3),
    user_id BIGINT NOT NULL,
    FOREIGN KEY (route_id) REFERENCES routes(id),
    FOREIGN KEY (user_id) REFERENCES users(id)
);
```

**Purpose:** Implements Many-to-Many relationship between users and routes

**Why Use a Join Entity Instead of @ManyToMany?**
- Allows storing additional information about the booking:
  - bookingDate: When the booking was made
  - seatNumber: Assigned seat
  - status: Booking status (CONFIRMED, PENDING, CANCELLED, COMPLETED)
- With direct @ManyToMany, we could only store the relationship

**Relationships:**
- MANY user_routes BELONG TO one user (Many-to-One)
  - Foreign Key: user_id → users.id
- MANY user_routes BELONG TO one route (Many-to-One)
  - Foreign Key: route_id → routes.id

## 4. IMPLEMENTATION OF SAVING LOCATION/ENTITY (2 Marks)

### How Data is Stored:

**Example: Saving a New User**

1. **Controller Layer** (`UserController.java`)
   ```java
   @PostMapping("/register")
   public ResponseEntity<User> registerUser(@RequestBody UserDTO userDTO) {
       User savedUser = userService.registerUser(userDTO);
       return ResponseEntity.ok(savedUser);
   }
   ```
   - Receives HTTP POST request with JSON data
   - Converts JSON to UserDTO object

2. **Service Layer** (`UserService.java`)
   ```java
   public User registerUser(UserDTO userDTO) {
       // Fetch the village from database
       Village village = villageRepository.findById(userDTO.getVillageId())
           .orElseThrow(() -> new RuntimeException("Village not found"));
       
       // Create user entity
       User user = User.builder()
           .name(userDTO.getName())
           .email(userDTO.getEmail())
           .phone(userDTO.getPhone())
           .userType(User.UserType.valueOf(userDTO.getUserType()))
           .village(village)  // Set relationship
           .build();
       
       return userRepository.save(user);
   }
   ```
   - Fetches related entity (Village) from database
   - Converts DTO to Entity
   - Sets up relationships
   - Calls repository to persist

3. **Repository Layer** (`UserRepository.java`)
   ```java
   @Repository
   public interface UserRepository extends JpaRepository<User, Long> {
       // save() method provided by JpaRepository
   }
   ```
   - Extends JpaRepository which provides CRUD operations
   - `save()` method persists entity to database

4. **Transaction Management**
   ```java
   @Transactional
   public class UserService { ... }
   ```
   - Ensures data consistency
   - If any step fails, entire operation is rolled back

### How Relationships are Handled:

**Foreign Key Management:**
- When saving User with village set, JPA automatically:
  1. Uses the village's ID
  2. Sets it as foreign key in users.village_id column
  3. No manual SQL needed!

**Cascade Operations:**
```java
@OneToMany(mappedBy = "province", cascade = CascadeType.ALL, orphanRemoval = true)
private List<District> districts;
```
- `cascade = ALL`: Save/Delete operations on Province propagate to Districts
- `orphanRemoval = true`: If a district is removed from the list, it's deleted from DB

## 5. SORTING AND PAGINATION IMPLEMENTATION (5 Marks)

### How Sorting is Implemented:

**Using Spring Data JPA's Sort Class:**

```java
// Service layer
public Page<User> getAllUsersPaginated(int page, int size, String sortBy, boolean ascending) {
    Sort sort = ascending ? Sort.by(sortBy).ascending() : Sort.by(sortBy).descending();
    Pageable pageable = PageRequest.of(page, size, sort);
    return userRepository.findAll(pageable);
}
```

**How it works:**
1. `Sort.by("fieldName")` creates sort configuration
2. `.ascending()` or `.descending()` specifies direction
3. Applied to query automatically by repository
4. Translates to SQL ORDER BY clause

**Example SQL Generated:**
```sql
SELECT * FROM users ORDER BY name ASC LIMIT 10 OFFSET 0;
```

### How Pagination Works:

**Using Pageable Interface:**

```java
Pageable pageable = PageRequest.of(0, 10, Sort.by("name").ascending());
Page<User> userPage = userRepository.findAll(pageable);
```

**What Happens Behind the Scenes:**

1. **Two Queries Executed:**
   ```sql
   -- COUNT query to get total records
   SELECT COUNT(*) FROM users;
   
   -- SELECT query with LIMIT and OFFSET
   SELECT * FROM users ORDER BY name ASC LIMIT 10 OFFSET 0;
   ```

2. **Page Object Contains:**
   - `content`: Actual data (List<User>)
   - `totalElements`: Total count
   - `totalPages`: Calculated based on page size
   - `number`: Current page number
   - `first`: true if first page
   - `last`: true if last page
   - `hasNext()`: Check if next page exists
   - `hasPrevious()`: Check if previous page exists

### Performance Benefits:

1. **Reduces Memory Usage:**
   - Only loads 'size' records instead of entire table
   - Example: 10 records vs 10,000 records

2. **Improves Response Time:**
   - Smaller result sets are faster to fetch
   - Faster serialization to JSON
   - Reduced network transfer

3. **Better UI Experience:**
   - First page loads quickly
   - Users can navigate through pages
   - Lazy loading on demand

4. **Database Optimization:**
   - Can use indexed columns for sorting
   - Reduces lock contention
   - Better query plan caching

**Endpoint Example:**
```
GET /api/users/paginated?page=0&size=10&sortBy=name&ascending=true
```

## 6. MANY-TO-MANY RELATIONSHIP (3 Marks)

### Implementation: User ↔ Route through UserRoute

**Why Not Direct @ManyToMany?**

Direct approach would be:
```java
// In User entity
@ManyToMany
@JoinTable(name = "user_route", ...)
private List<Route> routes;
```

**Problems:**
- Cannot store additional information (booking date, seat number, status)
- Limited flexibility

**Our Solution: Two Many-to-One Relationships**

```java
@Entity
public class UserRoute {
    @Id
    @GeneratedValue
    private Long id;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;
    
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "route_id")
    private Route route;
    
    // Additional fields
    private LocalDateTime bookingDate;
    private String seatNumber;
    private BookingStatus status;
}
```

### Join Table Structure:

```sql
CREATE TABLE user_routes (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,  -- Foreign key to users
    route_id BIGINT NOT NULL, -- Foreign key to routes
    booking_date TIMESTAMP NOT NULL,
    seat_number VARCHAR(255),
    status TINYINT,
    FOREIGN KEY (user_id) REFERENCES users(id),
    FOREIGN KEY (route_id) REFERENCES routes(id)
);
```

### Benefits:

1. **Stores Extra Information:** bookingDate, seatNumber, status
2. **Own Entity:** Can have its own repository, service, controller
3. **Flexibility:** Easy to add more fields later
4. **Query Flexibility:** Can query bookings directly

**Example Query:**
```java
// Find all confirmed bookings for a specific route
List<UserRoute> findByRouteAndStatus(Route route, BookingStatus status);
```

## 7. ONE-TO-MANY RELATIONSHIP (2 Marks)

### Examples in Our System:

**1. Province → Districts**

```java
// Province.java
@OneToMany(mappedBy = "province", cascade = CascadeType.ALL, orphanRemoval = true)
private List<District> districts;

// District.java
@ManyToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "province_id", nullable = false)
private Province province;
```

**Foreign Key Placement:**
- Stored in the "many" side: `districts.province_id`
- This is standard practice for One-to-Many relationships

**Other One-to-Many Relationships:**
- District → Sectors (foreign key: `sectors.district_id`)
- Sector → Cells (foreign key: `cells.sector_id`)
- Cell → Villages (foreign key: `villages.cell_id`)
- Village → Users (foreign key: `users.village_id`)

### How It Works:

**Saving with Cascade:**
```java
Province province = new Province();
province.setCode("01");
province.setName("Kigali City");

District district1 = new District();
district1.setCode("0101");
district1.setName("Gasabo");
district1.setProvince(province);

province.setDistricts(List.of(district1));

// Save province - districts are automatically saved
provinceRepository.save(province);
```

**SQL Generated:**
```sql
-- Insert province
INSERT INTO provinces (code, name) VALUES ('01', 'Kigali City');

-- Get generated ID (e.g., 1)

-- Insert district with foreign key
INSERT INTO districts (code, name, province_id) VALUES ('0101', 'Gasabo', 1);
```

## 8. ONE-TO-ONE RELATIONSHIP (2 Marks)

### Implementation: User ↔ DriverLicense

**User Entity (Owning Side):**
```java
@OneToOne(mappedBy = "user", cascade = CascadeType.ALL, orphanRemoval = true)
@EqualsAndHashCode.Exclude
@ToString.Exclude
private DriverLicense driverLicense;
```

**DriverLicense Entity (Owning Side):**
```java
@OneToOne(fetch = FetchType.LAZY)
@JoinColumn(name = "user_id", nullable = false, unique = true)
@EqualsAndHashCode.Exclude
@ToString.Exclude
private User user;
```

### Key Features:

1. **Foreign Key:** `driver_licenses.user_id`
2. **Unique Constraint:** Ensures one license per user
3. **Cascade:** When user is deleted, license is deleted
4. **Lazy Loading:** License loaded only when accessed

### How Entities Are Connected:

```java
// Create user
User user = new User();
user.setName("John");
user.setEmail("john@example.com");

// Create license
DriverLicense license = new DriverLicense();
license.setLicenseNumber("DL-123");
license.setExpiryDate(LocalDate.now().plusYears(5));
license.setUser(user); // Set relationship

// Save user (cascade will save license too)
userRepository.save(user);
```

**SQL Generated:**
```sql
-- Insert user
INSERT INTO users (name, email, ...) VALUES ('John', 'john@example.com', ...);

-- Get generated user ID (e.g., 1)

-- Insert license with foreign key
INSERT INTO driver_licenses (license_number, expiry_date, user_id) 
VALUES ('DL-123', '2031-01-01', 1);
```

## 9. existsBy() METHOD IMPLEMENTATION (2 Marks)

### Spring Data JPA Query Derivation:

**Repository Method:**
```java
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
    boolean existsByEmail(String email);
    boolean existsByPhone(String phone);
}
```

### How Existence Checking Works:

**Generated SQL:**
```sql
SELECT CASE 
    WHEN COUNT(u) > 0 THEN true 
    ELSE false 
END 
FROM users u 
WHERE u.email = ?
```

**Why More Efficient Than Fetching Entity?**

1. **Less Data Transfer:**
   - EXISTS: Returns single boolean
   - SELECT *: Returns all columns and data

2. **No Entity Creation:**
   - EXISTS: Just checks index/constraint
   - SELECT *: Creates full User object

3. **Faster Execution:**
   - Can stop at first match
   - Doesn't need to scan entire table

### Usage in Service:

```java
@Transactional(readOnly = true)
public boolean existsByEmail(String email) {
    return userRepository.existsByEmail(email);
}
```

### Controller Endpoint:

```java
@GetMapping("/exists/email/{email}")
public ResponseEntity<Map<String, Boolean>> checkUserExistsByEmail(@PathVariable String email) {
    boolean exists = userService.existsByEmail(email);
    Map<String, Boolean> response = new HashMap<>();
    response.put("exists", exists);
    return ResponseEntity.ok(response);
}
```

**Example Request:**
```
GET /api/users/exists/email/john@example.com
```

**Response:**
```json
{
    "exists": true
}
```

## 10. RETRIEVE USERS BY PROVINCE (4 Marks)

### Requirement:
Retrieve all users from a given province using EITHER:
- Province code (e.g., "01" for Kigali City)
- Province name (e.g., "Kigali City")

### Repository Method:

```java
List<User> findByVillageCellSectorDistrictProvinceCodeOrVillageCellSectorDistrictProvinceName(
    String provinceCode, 
    String provinceName
);
```

### Query Logic Explanation:

**Relationship Traversal Chain:**
```
User → Village → Cell → Sector → District → Province
```

**How Spring Data JPA Interprets the Method Name:**

1. `findBy` - Standard query derivation prefix
2. `Village` - Navigate to User.village property
3. `Cell` - Navigate to Village.cell property
4. `Sector` - Navigate to Cell.sector property
5. `District` - Navigate to Sector.district property
6. `Province` - Navigate to District.province property
7. `Code` - Access Province.code property
8. `Or` - OR logical operator
9. `VillageCellSectorDistrictProvinceName` - Same chain but access Province.name

**Generated JPQL:**
```jpql
SELECT u FROM User u 
WHERE u.village.cell.sector.district.province.code = :provinceCode
   OR u.village.cell.sector.district.province.name = :provinceName
```

**Generated SQL:**
```sql
SELECT u.* 
FROM users u
INNER JOIN villages v ON u.village_id = v.id
INNER JOIN cells c ON v.cell_id = c.id
INNER JOIN sectors s ON c.sector_id = s.id
INNER JOIN districts d ON s.district_id = d.id
INNER JOIN provinces p ON d.province_id = p.id
WHERE p.code = ? OR p.name = ?
```

### Alternative Implementation Using @Query:

```java
@Query("SELECT u FROM User u " +
       "JOIN u.village v " +
       "JOIN v.cell c " +
       "JOIN c.sector s " +
       "JOIN s.district d " +
       "JOIN d.province p " +
       "WHERE p.code = :code OR p.name = :name")
List<User> findUsersByProvince(@Param("code") String code, @Param("name") String name);
```

### Service Layer:

```java
@Transactional(readOnly = true)
public List<User> getUsersByProvince(String provinceCode, String provinceName) {
    return userRepository.findByVillageCellSectorDistrictProvinceCodeOrVillageCellSectorDistrictProvinceName(
        provinceCode, provinceName
    );
}
```

### Controller Endpoint:

```java
@GetMapping("/province/search")
public ResponseEntity<List<User>> getUsersByProvince(
    @RequestParam(required = false) String provinceCode,
    @RequestParam(required = false) String provinceName
) {
    String code = provinceCode != null ? provinceCode : "";
    String name = provinceName != null ? provinceName : "";
    
    List<User> users = userService.getUsersByProvince(code, name);
    return ResponseEntity.ok(users);
}
```

### Example Usage:

**By Code:**
```
GET /api/users/province/search?provinceCode=01
```

**By Name:**
```
GET /api/users/province/search?provinceName=Kigali City
```

**Both (OR logic):**
```
GET /api/users/province/search?provinceCode=01&provinceName=Kigali City
```

### Why This Design is Powerful:

1. **Flexible Querying:** Match by code OR name
2. **Single Method:** No need for multiple queries
3. **Automatic Joins:** Spring Data JPA handles complexity
4. **Type-Safe:** Compile-time checking of property names
5. **Performance:** Single SQL query with efficient joins
